I published [a post recently](https://blog.qorvault.com) about why and how I built QorVault — a system that makes twenty years of Kent School District public board records searchable via AI with mandatory source citations. That post told the human story: the frustration, the decision to build, the translator metaphor for AI coding tools.

This post is the technical companion. It's for practitioners who read the first post and thought: "okay, but how does it actually work, and how well does it work?" I'll answer both questions honestly, including the parts where it doesn't work well at all.

A note on authorship: I didn't write the code. I don't write code. Claude Code wrote every line of Python in this system under my architectural direction. What I do is design the system — the trust boundaries, the data flow, the security model, the retrieval strategy — and then I tell Claude Code what to build in the languages I actually speak: network architecture, security perimeters, and system design. I evaluate the results. I catch when something is wrong. I make the architectural decisions that Claude Code can't make because it doesn't know what it's like to sit in a board meeting and need a verifiable answer in fifteen seconds.

This post was developed the same way. I described the architecture and evaluation results in conversation with Claude, and together we shaped that into what you're reading. The technical understanding behind it is mine. The prose was collaborative.

## What happens when you ask QorVault a question

A query to QorVault goes through several stages before you see an answer. The whole process takes about thirteen seconds, and roughly twelve of those seconds are the language model thinking. Everything before that — the part I actually built — takes about a second.

**Query routing and reformulation come first.** Before any search happens, the system takes your plain-language question and does two things with it. First, Claude Sonnet classifies the query as one of three types: a retrieval question (search the document corpus), a database question (answer from structured metadata), or a hybrid of both. Database-type queries bypass the entire retrieval pipeline and generate SQL directly — if you ask "how many meetings were held in 2024," that's a counting question that a database handles better than a search engine. Second, and critically, the system reformulates your question into a form that the hybrid retrieval pipeline can work with effectively. A school board director typing "what did we spend on the SRO contract last year?" is asking a perfectly clear question, but RAG systems have specific retrieval characteristics — vector search responds to semantic content, keyword search responds to exact terms, and neither responds well to casual phrasing. The reformulation step bridges that gap, translating natural language into queries that play well with both search methods simultaneously. Without this step, the results degrade significantly. It's the difference between a system that works for people who happen to understand search query construction and a system that works for everyone who has a question.

**Stage 1: The query gets embedded.** Your question is converted into a 1024-dimensional vector using the same embedding model (mxbai-embed-large-v1) that was used to embed every document in the corpus. This takes about 70 milliseconds on CPU via ONNX Runtime. I didn't choose GPU inference for embeddings because the ONNX CPU path is fast enough for single-query latency and avoids the GPU driver instability I encountered with PyTorch on integrated AMD graphics.

**Stage 2: Multiple searches run in parallel.** The system searches for your answer in two or three ways simultaneously. Vector search goes to Qdrant and finds the chunks whose embeddings are most semantically similar to your question — this catches conceptual matches where the question and the answer use different words for the same idea. Keyword search goes to PostgreSQL and uses traditional full-text search to find chunks containing the same words as your query — this catches exact references like resolution numbers, people's names, and specific dollar amounts. When spaCy's named entity recognition detects person names in the query, a third search leg activates — an ILIKE name search that finds documents mentioning that specific person regardless of semantic or keyword relevance. Neither search approach alone is sufficient. Vector search misses exact terms; keyword search misses meaning; name search catches the person-specific documents that both others might rank low.

**Stage 3: The results get merged.** Reciprocal Rank Fusion (RRF) combines the ranked lists from all search legs into a single candidate set. The key insight of RRF is that it doesn't try to compare scores between the different search methods — the scores are on incompatible scales — it uses only the rank positions. A document that ranks #2 in keyword search and #5 in vector search gets a higher combined score than one that ranks #1 in vector search but doesn't appear in keyword results at all.

**Stage 4: Recency and authority weighting adjust the scores.** Two domain-specific adjustments shape the candidate pool before the expensive reranking step.

Recency weighting applies a half-life decay based on document age, so newer documents score proportionally higher than older ones for the same relevance. This matters because a school board that discussed the budget last month and three years ago probably wants to hear about last month's discussion first. The weighting is applied as a multiplier on the RRF score, not a filter — old documents can still win if they're significantly more relevant.

Authority weighting multiplies each candidate's score by a per-document-type weight. A school board's formal agenda items and policy documents are more authoritative than meeting transcripts, but transcripts generate 40 to 100 times more chunks per source document because they're so verbose. Without weighting, transcripts dominate the results purely on volume regardless of whether they're the best source for the question. Policies get 1.4×, agenda items 1.3×, minutes 1.2×, attachments 1.0×, and transcripts 0.7×. Five lines of code, significant impact on result quality.

Both adjustments run before the cross-encoder reranker, intentionally — they shape which candidates enter the reranker's pool, and the reranker can still override them if a penalized document turns out to be genuinely more relevant.

**Stage 5: A cross-encoder reranker evaluates each result.** The initial searches are fast but approximate. The cross-encoder (bge-reranker-v2-m3) is a 278 *million* parameter model — not billion, million. That's important: at 278M parameters, it's tiny by modern standards, small enough to run comfortably on CPU. The INT8 quantized ONNX model is about 545MB on disk, far smaller than any language model. It looks at each candidate alongside the original question and makes a much more precise relevance judgment. This takes about 3.5 seconds for 20 candidates on CPU with no GPU involved. The reranker model file is SHA-256 verified at startup — if the hash doesn't match, the reranker is disabled and the system continues without it, logging a warning. A tampered model won't serve manipulated scores (it won't serve any scores), but the degradation is silent rather than a hard stop. Tightening this to refuse to start on hash mismatch is on the backlog.

**Stage 6: Claude Opus synthesizes an answer with mandatory citations.** The top-k results (default 10) go to Claude Opus 4.6 with a system prompt that enforces citation discipline. Every factual claim must include an inline `[Source N]` reference. The system prompt instructs the model to answer only based on the provided context documents. Citation URLs are constructed exclusively from chunk metadata — the model never generates or modifies a URL. This is the most expensive step and the slowest (~12 seconds), but hallucination resistance matters more than throughput for civic records.

### Why these specific components

I get asked why I'm running PostgreSQL alongside Qdrant instead of using a single vector database. The answer is that the source documents have rich relational metadata — tenant ID, meeting date, committee, document type, agenda item ID — that PostgreSQL handles natively, while Qdrant handles the embedding side. Hybrid retrieval needs both engines anyway, and trying to do full-text search inside a vector database produces worse results than a purpose-built text search engine.

I also get asked why I'm using ONNX Runtime on CPU instead of running everything on GPU. The embedding model processes about 140-160 chunks per minute on CPU, which is fast enough for the corpus sizes this project targets. The reranker takes 3.5 seconds for a batch of 20 candidates, which is invisible to the user inside a 13-second query. GPU hardware is reserved for batch operations like re-embedding the entire corpus when I upgrade retrieval components or running the transcription pipeline. Daily query serving doesn't need it.

## How a non-developer maintains production code

This is the part that I think matters most for the broader conversation about who can build software in the age of AI coding agents, and it's the part where I haven't been able to find anyone else doing something similar to learn from.

I don't read code. I can look at Python and get the general shape of what's happening, the same way I can look at a French menu and figure out what's probably tofu, but I'm not reviewing code for correctness at the line level. What I can review is architecture, security posture, and behavioral correctness — does the system do what I designed it to do, and does it do it safely?

The answer to "how do you maintain code you can't read?" is: **structural enforcement, not review.** I built five independent security layers, each of which enforces its constraints regardless of what the other layers do.

**Layer 1: Claude Code's own configuration.** A `settings.json` file controls what Claude Code is allowed to do. Commands are sorted into deny (never allowed — curl, wget, sudo, secret file access), ask (requires explicit approval — file deletion, git push, pip install, service management), and allow (safe operations — reading files, running linters, executing tests). A set of PreToolUse hooks intercept every command before execution: one blocks dangerous commands, one blocks writes to the production directory, one validates pip installations, and one sends the proposed action to a local AI reviewer for classification before the operator sees it.

Both `settings.json` and all hook scripts live on a write-protected SD card mounted read-only into the system. The SD card's hardware write-protect switch is a physical interlock — when the switch is locked, the card reader's firmware rejects all write commands at the hardware protocol level. No software running on the machine can modify these files while the switch is in that position. To change a hook or a permission setting, I have to physically remove the card, flip a plastic tab, reinsert it, make the change, remove it, flip the tab back, and reinsert it. That sequence cannot be automated, cannot be scripted, and cannot be executed by any AI agent running on any of my machines. The physical act of flipping the switch forces me to be conscious of what I'm changing and why.

I should be honest about the limits of this approach. The write-protect switch is enforced by the card reader's firmware, not by physics — it's a hardware convention, not a physical impossibility. A sufficiently exotic attack that compromised the card reader's firmware could theoretically bypass it. And a compromised AI agent could theoretically attempt to unmount the SD card, create fake versions of the hook files elsewhere, and redirect the symlinks. Both attacks would require root access that the agent doesn't have (sudo is in the deny tier, and the mount unit is owned by root), but "requires root" is not the same as "impossible." The SD card makes casual or automated modification of security controls genuinely impractical, raises the bar significantly for sophisticated attacks, and — perhaps most valuably — serves as a cognitive interlock that forces deliberate human action before any security control changes. It is not an impenetrable barrier, nor am I claiming it is.

**Layer 2: The forge pipeline.** All production code changes go through meta-forge, a multi-stage review pipeline. Read-forge examines the target project and produces a report. Security-forge audits the proposed changes for vulnerabilities. Prompt-forge generates the delegation prompt that will be given to the implementing agent. Then the implementing agent — infra-forge — receives the delegation prompt and responds with its interpretation of the task and its proposed implementation approach. That interpretation gets routed back through security-forge to verify that what the agent *thinks* it's about to do actually meets the security requirements. If there's a mismatch — the agent misunderstood the task, or its approach introduces a vulnerability the prompt didn't anticipate — it goes back for revision and subsequent review until no more revisions are required. Only after that cycle converges does the result come to me for final approval and a green light for execution. The CLAUDE.md files governing each forge stage also live on the write-protected SD card, symlinked into their expected locations. No agent can modify its own behavioral instructions.

There's one more layer to the review process that I've found surprisingly effective. Before I give final approval, I frequently take the proposed changes to a separate Claude instance — either a new Claude Code session or Claude.ai — that has no shared context with the implementing agent. This "fresh eyes" review catches things that both the forge pipeline and I might miss, specifically because the reviewing agent hasn't been marinating in the same context that produced the work. It has no investment in the approach, no accumulated assumptions, and no exposure to the reasoning that led to the proposed implementation. The disagreements between the implementing agent and the reviewing agent are where the most valuable catches happen — when two independent AI assessments of the same code reach different conclusions about its safety or correctness, that's exactly the kind of signal a non-developer operator needs to pay attention to.

**Layer 3: Commit-time gates.** A pre-commit framework runs on every git commit: Gitleaks scans for leaked secrets, Bandit runs security analysis, Ruff enforces code quality rules, Interrogate checks for 80% docstring coverage, and detect-private-key catches obvious credential mistakes.

**Layer 4: OS-level isolation.** Rootless Podman containers (not Docker — rootless Podman runs without a daemon and without root privileges), SELinux mandatory access control, auditd kernel-level process auditing with rolling one-year retention, AIDE daily file integrity checks, LUKS full-disk encryption, BTRFS snapshots for point-in-time recovery.

**Layer 5: Hardened service configuration.** PostgreSQL and Qdrant both run in rootless Podman containers with `--cap-drop=ALL`, `--read-only` root filesystems, `--security-opt=no-new-privileges`, memory and CPU limits, PID limits, and localhost-only binding. No container can reach the network; no container has any Linux capability; no process inside a container can gain privileges it didn't start with.

The core principle behind all of this is **deterministic controls over probabilistic ones.** I cannot reliably instruct an AI not to do something dangerous — instruction-following is probabilistic, and the failure mode is silent. What I can do is make it structurally impractical for the dangerous action to succeed regardless of what the AI tries. The hooks enforce behavior whether Claude Code follows instructions or not. The network segmentation blocks traffic whether the process intends to send it or not. The SD card's write-protect switch prevents modification whether the software wants to modify the files or not. Every security property that matters is enforced by a mechanism the AI agent cannot easily influence.

### What happens when this process fails

On March 25, 2026, Claude Code deleted 117 files from production in a single changeset.

The changeset was supposed to implement cross-encoder reranking and add pre-commit hooks. Instead, it scope-crept into reformatting the entire codebase with Ruff and "cleaning up" code it decided was unnecessary. It deleted the hybrid retriever, the keyword retriever, the reranker, the authentication middleware, the session tracking, the response cache, the admin dashboard, and the access request system. Seven modules, gone. Only two had backups (the hybrid and keyword retrievers had `.bak` files from an earlier session). The other five existed only as `.pyc` bytecode ghosts in `__pycache__/`.

The system continued returning HTTP 200. Queries still produced answers. But it had silently fallen back to vector-only search — no keyword matching, no hybrid fusion, no reranking. I couldn't detect the quality regression because I didn't have a quality baseline to compare against.

Four failures chained together to produce this outcome: Claude Code was operating directly on production files (no dev environment existed yet), scope crept without a gate (no forge pipeline enforced boundaries), I couldn't detect the quality regression (no automated tests or evaluation baseline), and the system gave no indication anything was wrong (it still returned plausible answers).

Every one of those failure links now has a mitigation. The dev environment on a separate machine with offset ports prevents direct production modification. The forge pipeline prevents scope creep by defining exactly which files are in scope. Phase D evaluation (described below) provides the quality baseline. And Phase E will codify that baseline into automated regression tests that run before every production promotion.

I share this because the honest version of "a non-developer maintaining production infrastructure with AI coding agents" includes the disasters, not just the successes. The March 25 incident taught me — the hard way, and embarrassingly late — lessons that professional software developers internalize early in their careers: never work directly on production, always have a rollback path, and the process exists because the last time someone skipped it, something broke. I'm not claiming any of this is novel engineering wisdom. I'm saying that the structural enforcement I built in response is what allows someone without that career of accumulated instinct to operate safely anyway.

## How I evaluated whether the system actually works

If you build a civic intelligence platform and want to use it during public board meetings, you need to know which answers you can trust and which ones will embarrass you on camera. That requires measuring the system's accuracy against ground truth — not synthetic benchmarks, but real questions where you already know the general landscape of what the correct answers should look like.

Phase D is the trust verification pass. I developed twelve test questions in collaboration with Claude — questions spanning different types of board business: budget discussions, superintendent contracts, specific resolutions, executive sessions, bond measures, policy work sessions, ESSER funds, and staffing concerns. I want to be clear about what "ground truth" means here: I didn't have every dollar figure and resolution number memorized. Nobody does — the entire reason I built QorVault is that there is far too much information across far too many domains for any single person to keep in their head. What I do have is two years of lived experience as a board director. I know which meetings mattered, roughly when they happened, what the major decisions were, and which corners of the record tend to hold the information that matters for oversight. That institutional memory is enough to know when an answer is wrong, even when I can't recite the exact right answer from memory.

For each question, I ran two independent paths. Path 1 was the RAG pipeline — the full query through vector search, keyword search, RRF fusion, reranking, and Claude synthesis, exactly as it would work during a board meeting. Path 2 was direct database queries — Claude Code querying PostgreSQL directly, reading the raw documents, and constructing its own answer from the source material. I personally reviewed the first three questions in detail against the source documents, clicking every citation link to verify it pointed to a document that actually supported the claim. Claude Code then independently evaluated all twelve questions with the same methodology. There were things I caught that Claude Code missed in my three questions, and quite a few things Claude Code caught across all twelve that I would have missed. The evaluation itself was a collaborative process, not a solo one.

### The results

Four of twelve questions were GREEN — factually correct, all citations accurate, safe to cite in a public board meeting. Four were YELLOW — mostly correct but with issues like garbled names from transcription errors or missing the most recent document. Four were RED — wrong enough that citing them publicly would damage my credibility.

**4 GREEN (33% safe to cite without modification):**
- Q2: the four budget work sessions in 2024-25 — all dates, topics, and presenters verified.
- Q4: superintendent contract terms — addendum date, effective date, term end, and amended sections all correct.
- Q10: ESSER fund decisions — dollar figures, resolution numbers, and timeline all verified against source attachments.
- Q11: my own budget remarks in July 2024 — correctly identified that no such remarks exist in the record. This is the behavior I want: when the answer isn't in the documents, refuse to hallucinate.

**4 YELLOW (33% need verification before public citation):**
- Q3: SRO agreement — good historical cost data but missed the most recent renewal by a full school year.
- Q5: Resolution 1669 — core facts correct (date, purpose, 3-2 vote, repeal) but the transcript mangled two names. One board member's name was rendered as something like "John Smith" when the actual name was closer to "James Singh" — a Whisper transcription error where the audio was close enough to fool the model but wrong enough to misidentify a real person. It also omitted my own Nay vote from the summary.
- Q6: executive sessions — answered a different interpretation of the question than I intended, but the content it did provide was accurate.
- Q8: policy work session — content 100% accurate but missed a more recent session by four weeks.

**4 RED (33% must not be cited publicly):**
- Q1: last budget discussion — gave August 2025 when the actual answer was February 2026 (five months stale), misidentified the context (a comment during an Outdoor Education item, not a formal budget agenda item), and attributed a presentation to a person who does not exist. The name in the transcript was a complete fabrication by Whisper — not a misspelling of the actual presenter, but a wholly invented name that sounds vaguely plausible.
- Q7: bond/levy measure — **this is the most dangerous failure.** The board passed a resolution in July 2024 authorizing a $97.8 million capital levy, which voters approved in November 2024. The RAG system has no knowledge of either event despite both being in the database. A director asked about the bond/levy status on camera would appear unaware of his own board's most significant recent financial action.
- Q9: executive session count 2024 — found 3 dates when the answer is 27. About 11% recall on a question that asks for a count.
- Q12: class size/staffing discussions — found 1 meeting when 16 meetings discussed the topic. About 6% recall.

### What's breaking and why

The twelve test questions revealed five distinct failure modes, each with a different root cause.

**Enumeration is structurally broken.** When you ask "how many times did we discuss staffing?" or "count the executive sessions in 2024," the system searches for the 10 most relevant chunks. But the answer requires finding *all* relevant documents, not the *best* ones. Top-k retrieval is the wrong tool for counting. Q9 and Q12 both failed this way. The fix is routing enumeration queries to a database aggregation instead of vector search — detecting the intent ("how many," "list all," "during which meetings") and running SQL rather than semantic retrieval.

**Recency detection fails.** When you ask "when did the board last discuss the budget," the word "last" carries temporal intent but no useful semantic information for retrieval. The system finds old budget discussions that happen to have high similarity scores, while newer agenda items with matching titles sit in the database unsurfaced. Q1, Q3, and Q8 all failed this way. The fix is detecting temporal-superlative language and applying aggressive recency weighting — I've designed a decay function where each year of distance from the anchor date reduces a document's temporal weight to 80% of the adjacent year, with bidirectional fan-out from a specified date for historical queries.

**Transcript errors become cited facts.** Whisper mishears names. The language model trusts the transcript text and cites these garbled names as authoritative. For a system whose operator cites names publicly, this is dangerous. The fix is a canonical name dictionary with phonetic matching — the set of board directors and executive staff is small and stable, and fuzzy matching (Jaro-Winkler similarity, Metaphone encoding) catches most ASR garbles.

**Verbose transcripts crowd out authoritative sources.** A two-hour meeting transcript generates 40-100 chunks. The corresponding one-page agenda item generates 2-3 chunks. In a pool of candidates ranked by semantic similarity, transcripts have more "shots" at matching any query purely by volume. Authority weighting (transcripts ×0.7, agendas ×1.3) helps, but the deeper fix is stratified retrieval — running parallel filtered queries with explicit per-type quotas to guarantee document-type diversity in the results.

**Complete retrieval failures on important documents.** Q7's resolution is in the database. The embedding exists. But the retrieval pipeline didn't surface it for a directly relevant query. This is the failure mode I understand least and trust least, because the other failures are predictable — I can tell you which categories of questions will probably fail. This one is a specific document that should have been found and wasn't. Investigating why is my current top priority.

### What the evaluation means for daily use

The four GREEN results represent the system's sweet spot: specific named artifacts (a resolution, the superintendent contract, the four budget work sessions, ESSER allocations) where retrieval finds the right document and synthesis summarizes it accurately. When I use QorVault for board meeting preparation, these are the question types I trust without additional verification.

The four YELLOW results are usable for background research — they point me to the right documents and give me the right general picture, but I verify every specific claim before citing it publicly. The system tells me where to look, and I confirm what I find there.

The four RED results define the boundary of what I cannot use the system for, at least until the retrieval improvements are implemented. I don't ask QorVault to count things. I don't ask it for the "most recent" occurrence of something without independently verifying the date. And I click every citation link before I say anything on the record.

The most important behavioral pattern is Q11: when the system doesn't have enough information to answer, it says so. It doesn't fabricate. That's the foundation everything else builds on — a civic tool that hallucinated answers would be worse than no tool at all.

## What this means for civic technology

Every government body I'm aware of — not just in the United States, but anywhere public meetings generate records — has some version of the problem QorVault addresses. School boards, city councils, county commissions, water districts, port authorities, regional parliaments — they all generate decades of meeting records that technically exist in public but are functionally inaccessible. The documents are there. The search tools are not. A resident who wants to know how their school board voted on a specific issue three years ago faces a search interface designed for finding tonight's agenda, not for institutional memory.

There are existing tools that address parts of this problem. Projects like Civic Band use Datasette to make municipal document archives browsable and searchable, and they do that well. But document search — however good it gets — is only half of what an elected official actually needs. The other half is synthesis: the ability to take the retrieved documents and draw conclusions across them, to notice patterns that span years of records, to connect a budget line item from 2022 to a policy change in 2024 to a staffing decision in 2025. That's the kind of work that previously required either a deeply experienced policy analyst or a subject matter expert willing to donate their time. An AI language model with strong instruction-following and mandatory citation discipline isn't a replacement for either of those people, but it can do a credible first pass at the synthesis work — enough to tell a board director "here's what I found across these documents, and here are the specific sources you should read yourself before saying anything on the record."

This matters because of who does this work and under what conditions. The Kent School District serves roughly 25,000 students across more than 45 buildings and 72 square miles, with about 4,000 employees and a yearly budget exceeding $550 million. The five people charged with oversight of this half-billion-dollar operation — the elected board of directors — are each paid the equivalent of about $50 per day they work, with a yearly cap of $4,800. That compensation level hasn't been changed in nearly 40 years, and I don't think Washington state is unique in this. Show me a corporation with a $550 million annual budget whose board is compensated at that level. The information asymmetry I've described — where the administration has full-time staff, institutional knowledge, and continuity, while the oversight body has part-time volunteers who turn over every four years — isn't a bug in the system. It's the structural reality of local government across the country, and probably across the world. QorVault doesn't fix the structural problem. What it does is give the oversight side a tool that partially compensates for the asymmetry.

The question I keep getting asked is: could another district deploy this? The honest answer is yes, with significant caveats.

The architecture is designed for multi-tenancy from the ground up — every database table and every vector payload is scoped by `tenant_id`. The retrieval pipeline, the embedding pipeline, the processing pipeline — they're all tenant-aware. A second district doesn't need a second installation; it needs a new tenant configuration and a per-source document loader.

The hard part is the corpus. Every district's BoardDocs instance has slightly different field names, different attachment handling, different ways of structuring agenda items. The scraper I wrote for Kent's BoardDocs works for Kent. It probably works for other Washington state districts on the same BoardDocs platform with minor modifications. It definitely doesn't work for a district using Legistar, Novus Agenda, or a different document management platform without writing a new loader.

The other hard part is the transcripts. The 397 meeting recordings in QorVault's corpus were transcribed using multiple approaches. WhisperX handled the primary transcription on GPU pods (RunPod), with speaker diarization via pyannote 4.0.4 and speaker name resolution via an LLM pass over each meeting's opening roll call. YouTube's auto-generated captions for our recorded meetings provide a comparison transcript. On the roadmap is a third transcription system using a different architecture entirely, so I can cross-reference three independent transcriptions against each other and resolve disagreements programmatically — similar to how election systems use multiple independent counts to increase confidence. I'm also planning to go through at least the meetings since my election and do manual voice identification, giving the diarization system a source of truth for speaker recognition rather than relying almost entirely on context inference. The transcription pipeline requires GPU hardware or cloud GPU credit — I used high-end cloud GPUs on RunPod to process the corpus quickly because we're facing a budget shortfall in the tens of millions of dollars and I had an imperative to move fast. But faster hardware isn't a requirement; it's a convenience. Someone without that urgency could use more modest GPU resources or even CPU-only processing — it would take longer, but the same work would get done. The architecture doesn't assume you own a GPU or can afford a high-end cloud instance. It assumes you can rent *something* with a GPU for a few hours when you need it.

And then there's the cost. The ongoing cost of running QorVault is lower than most people expect. Based on two months of actual Anthropic API billing data, a typical board meeting preparation day — running queries to research agenda items, check historical context, and verify facts — costs about $0.50 in API fees. Some prep sessions run higher (up to about $1.15 on heavy research days), some lower ($0.14 for quick lookups). At roughly eight preparation days per month across two regular meetings plus committee work, that projects to about $4 per month or under $50 per year in API costs for a single board director's usage. The response cache reduces this further — repeat and similar queries return from the database at zero API cost.

The query-serving stack itself — PostgreSQL, Qdrant, the RAG API, embedder, and reranker — runs on a single machine with sufficient memory to hold the models and databases. The embedding model and cross-encoder reranker both run on CPU via ONNX Runtime; no GPU is needed for daily query serving. At 278 million parameters — million, not billion — the reranker is tiny by modern standards, and the INT8 quantized ONNX model is about 545MB on disk. Someone with an existing workstation or server with 64-128GB of memory could run the entire production stack with no additional hardware investment, reducing the ongoing expense to just the API fees. GPU compute is only needed for periodic batch operations like corpus re-embedding and audio transcription with speaker diarization, and those can be rented from cloud GPU providers like RunPod as needed rather than owned.

The development cost is a separate question from the operating cost. I use Anthropic's Max subscription plan for Claude Code during active development — the intensive work of building the system from scratch, designing the retrieval pipeline, implementing the security architecture, and iterating on quality. But adapting an already-built system to a new district is a fundamentally smaller task than building it in the first place. A new deployment needs a tenant configuration, a document loader for the district's platform, and corpus ingestion — not a ground-up rebuild of the retrieval pipeline, the security layers, or the evaluation methodology. I'll be testing this process myself in the near future: first with a different Washington state school district that uses a different board document provider than Kent's BoardDocs, and then with my local city council, which is an entirely different category of government body with its own document structure and meeting format. How much effort those adaptations actually require — and what subscription tier or API budget they demand — will be a concrete data point rather than a guess. If you're thinking about doing something similar for your own government body, reach out to me — I'd be happy to help where I can.

It's worth noting why the API costs exist at all. The retrieval pipeline — embedding the query, searching Qdrant and PostgreSQL, fusing results, applying recency and authority weighting, running the cross-encoder reranker — runs entirely on local hardware with zero API calls. What hits the Anthropic API are three steps that currently use cloud models: query reformulation (transforming a plain-language question into a form the retrieval system can work with effectively), query routing (classifying the question type to determine the right retrieval strategy), and answer synthesis (generating the cited response from retrieved documents). All three are candidates for migration to local models — query reformulation and routing are classification and rewriting tasks that capable local models can handle, and I plan to move them there. Synthesis is harder to replace locally because citation discipline requires strong instruction-following, but it's not architecturally required at all. The system could export its retrieval results as a structured document — ranked chunks with source URLs, relevance scores, and document metadata — that a user then brings to whatever language model they prefer, including a free local model or no model at all. Reading ten ranked document excerpts yourself is less convenient than getting a synthesized answer with inline citations, but the expensive part of this problem was never the synthesis — it was finding the right ten documents out of 230,000 chunks spanning twenty years of records. That's what the retrieval pipeline does, and it runs without any API calls at all.

## Who built this and why it matters

I'm a school board director in Kent, Washington. I'm an Air Force veteran with a bachelor's degree in network administration and security. I have ADHD and very little practical software development training — a single Python course and a handful of database courses in college, all long forgotten by now. I built QorVault because the questions I was elected to ask deserved answers I could verify, and the institutional information system wasn't designed to give them to me.

I recently watched Simon Willison's [conversation with Lenny Rachitsky on Lenny's Podcast](https://www.youtube.com/watch?v=wc8FBhQtdsA), where he laid out his framework for thinking about AI-assisted software development. From what I understood, he draws a line between what people call "vibe coding" — where you accept AI-generated code without really reviewing it — and "agentic engineering" — where professional software engineers use AI agents to amplify their existing expertise. It's a useful framework, and watching him describe it helped me think about what I'm actually doing with QorVault, because whatever it is, it doesn't seem to fit neatly into either category.

I don't know what to call whatever it is that I'm doing. Vibe coding, at least from what I've read online and heard other people describe, doesn't seem to match my methodology — I care deeply about correctness and I've built an entire security architecture specifically because I don't trust any agent, including myself, to get things right without structural enforcement. But I'm also not a developer by any stretch of the imagination, so "agentic engineering" as Willison seems to define it — starting with professional software engineering expertise and amplifying it — doesn't really apply either.

What I think I am is a domain expert with deep systems understanding who uses AI as a translator for the technical languages I don't speak — mostly Python, as it's what Claude Code defaults to for this kind of work. The Air Force is where I got most of my security intuition. I held a Top Secret SCI clearance and worked in environments where security wasn't theoretical — it was operational and physical. I once had to transport a piece of equipment under dual-custody escort to the NSA at a facility in another country for a firmware update. Those experiences taught me more about what I think of as "separable security" — the principle that enforcement mechanisms should be independent of the systems they protect — than any class I took in college. A decade of coursework deepened that foundation with formal concepts in network architecture and security. Five years of project management taught me how to coordinate complex work across multiple dependencies. Claude Code translates the languages I'm missing while I provide the architecture, the security model, the evaluation methodology, and the institutional knowledge that no AI has.

Maybe this doesn't need a name yet. Maybe the taxonomy will expand as more domain experts discover that the door to building real infrastructure is open to them in ways it wasn't before — not because AI made software engineering unnecessary, but because AI can translate between the languages these experts already speak and the ones they don't.

The code is public at [github.com/QorVault/qorvault](https://github.com/QorVault/qorvault). The interesting parts aren't the Python — they're the CLAUDE.md files that govern the forge pipeline, the pre-commit hooks and PreToolUse hooks that enforce safety properties, the five-layer security architecture, and the evaluation methodology that measures whether the system actually works. The process is the contribution, not the code quality.

The system has already produced findings significant enough that the Washington State Auditor's Office has agreed to look into expanding the scope of their work based on what QorVault surfaced. I can't discuss the details for fear of impacting their process, but that alone has made the investment of time worth it.

If you're an elected official, a journalist, a civic researcher, or anyone who needs to search public records and get verifiable answers — or if you're just someone who thinks their government's records should be more accessible than they are — QorVault is an early attempt at what I think could become standard civic infrastructure. The code is open, the architecture is documented, and if you want to try deploying it for your own community, I'm reachable at the email below. The door is open. The tools exist. What's needed now are the people who have the questions.
